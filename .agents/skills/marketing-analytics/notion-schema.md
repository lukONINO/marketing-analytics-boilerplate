# Notion — Database Schema & Page Templates

Defines the exact shape of the **Marketing Reports** database in
Notion, the properties each page must carry, and the body block
structure Claude composes for daily + weekly reports.

## Database location

**Parent page**: `<your Notion parent page>` (e.g. "GEO/SEO Analytics"
under your Marketing or Growth workspace).

Parent page ID (for MCP calls): `<placeholder>`

The database ID itself is written to `config/notion_schema.yaml`
after the first-run bootstrap creates it.

---

## First-run bootstrap

Runs once, only when `config/notion_schema.yaml` has `database_id: null`.

### Step 1 — Confirm the parent page is accessible

```
notion-fetch id="<placeholder>"
```

If this fails with "not found / not accessible", the Notion MCP
integration isn't shared with the parent page. Tell the user:

> *"The Notion integration doesn't have access to the parent page. Open
> that page in Notion → click `...` (top right) → Connections → add
> your integration. Then re-run."*

### Step 2 — Create the database

Use `notion-create-database` with the schema below (SQL DDL per the MCP's format):

```sql
CREATE TABLE (
  "Name" TITLE,
  "Type" SELECT('Daily':blue, 'Weekly':purple),
  "Period start" DATE,
  "Period end" DATE,
  "SEO score" NUMBER,
  "GEO score" NUMBER,
  "LLM traffic sessions" NUMBER,
  "Total organic clicks" NUMBER,
  "Total sessions" NUMBER,
  "Conversions" NUMBER,
  "Key insights" RICH_TEXT,
  "Anomalies" MULTI_SELECT(
    'Traffic spike':green,
    'Traffic drop':red,
    'Rank drop':red,
    'Rank climb':green,
    'New competitor surfaced':orange,
    'GEO opportunity':yellow,
    'CRO opportunity':yellow,
    'Content gap':yellow,
    'Coverage gap':gray,
    'Data quality flag':gray
  ),
  "Status" SELECT('Generated':gray, 'Reviewed':blue, 'Actioned':green),
  "Generated by" SELECT('claude-daily':blue, 'claude-weekly':purple, 'manual':gray),
  "Source data file" RICH_TEXT,
  "Generated at" CREATED_TIME
)
```

MCP call:

```
notion-create-database(
  parent={"page_id": "<placeholder>", "type": "page_id"},
  title="Marketing Reports",
  description="Daily digest + weekly deep-dive reports generated by the marketing-analytics skill.",
  schema=<the CREATE TABLE above>
)
```

### Step 3 — Persist the IDs

The `notion-create-database` response includes a data source ID in a
`<data-source url="collection://...">` tag and a database ID. Write
both to `config/notion_schema.yaml`:

```yaml
marketing_reports:
  # (parent_page_* unchanged)
  database_id: "<returned database UUID>"
  database_url: "<returned page URL>"
  data_source_id: "<UUID from collection:// URL>"
```

### Step 4 — Sanity-fetch the database

```
notion-fetch id="<database_id>"
```

Confirm the 14 properties are all present. If any are missing,
`notion-update-data-source` is your fix path — don't recreate the DB
(you'd lose the ID).

## Property → aggregator field mapping

When composing a page, populate properties from the aggregate JSON as:

### Daily (`data/processed/daily/<date>.json`)

| Property | Source field |
|---|---|
| `Name` | `"Daily Report – " + date` |
| `Type` | `"Daily"` |
| `Period start` | `date` |
| `Period end` | `date` |
| `SEO score` | `scores.seo_score` |
| `GEO score` | `scores.geo_score` |
| `LLM traffic sessions` | `summary.llm_traffic.sessions` (0 if not available) |
| `Total organic clicks` | `summary.seo.total_clicks` |
| `Total sessions` | `summary.traffic.sessions` |
| `Conversions` | `summary.traffic.conversions` |
| `Key insights` | Claude-composed 3-5 bullets (plain text, newline-separated) |
| `Anomalies` | Multi-select tags derived from `deltas_vs_prior_*` + `summary.geo.data_quality_flags` |
| `Status` | `"Generated"` |
| `Generated by` | `"claude-daily"` (or `"manual"`) |
| `Source data file` | `"data/processed/daily/" + date + ".json"` |

### Weekly (`data/processed/weekly/<iso_week>.json`)

| Property | Source field |
|---|---|
| `Name` | `"Weekly Report – Week " + week_number + " " + year` |
| `Type` | `"Weekly"` |
| `Period start` | `start_date` |
| `Period end` | `end_date` |
| `SEO score` | `week_summary.avg_seo_score` |
| `GEO score` | `week_summary.avg_geo_score` |
| `LLM traffic sessions` | `week_summary.total_llm_sessions` |
| `Total organic clicks` | `week_summary.total_clicks` |
| `Total sessions` | `week_summary.total_sessions` |
| `Conversions` | `week_summary.total_conversions` |
| `Key insights` | Exec summary bullets (5 max) |
| `Anomalies` | Multi-select from `anomalies[*].metric` |
| `Status` | `"Generated"` |
| `Generated by` | `"claude-weekly"` |
| `Source data file` | `"data/processed/weekly/" + iso_week + ".json"` |

## Daily page body

Composed as real Notion blocks via `notion-create-pages`. Order:

1. **H1**: `Daily Report – <date>`
2. **Callout** (gray background, chart icon): Key metrics summary
   ```
   SEO score: <n>  |  GEO score: <n>  |  LLM sessions: <n>
   Organic clicks: <n>  |  Sessions: <n>  |  Conversions: <n>
   Deltas vs yesterday: <readable summary, e.g. "sessions -8%, clicks +12%">
   ```
3. **H2**: `SEO Performance`
   - One paragraph narrative (what's notable today)
   - Bulleted highlights (3-5 bullets: avg position, top query, biggest change, etc.)
   - **Table**: top 5 queries (query / page / clicks / impressions / position)
   - **Table**: top 5 pages (page / clicks / impressions / CTR)
4. **H2**: `GEO (AI Visibility)`
   - Paragraph narrative from Peec data — who's winning, who's losing
   - Active engines count with warning about inactive ones
   - **Table**: brand leaderboard (brand / visibility / SoV / sentiment)
   - Top 3 most-cited URLs from Peec
5. **H2**: `LLM Referral Traffic`
   - Paragraph narrative
   - Providers hit today (bulleted)
   - Top landing pages for LLM traffic (if any)
   - Note if 0 sessions: "Zero LLM-referral sessions today — see
     cross-channel note below about AI Overview attribution."
6. **H2**: `Cross-Channel Analysis` ← REQUIRED
   - Paragraph explicitly linking SEO × GEO × LLM findings
   - Minimum one observation spanning ≥2 channels
   - If sample size is too small, SAY SO
7. **H2**: `Anomalies & Alerts` (omit if zero items)
   - Bulleted list, each with metric + value + direction + sigma
8. **H2**: `Recommended Actions` (daily: 1-3 items, concrete)
9. **Toggle**: `Raw data references`
   - `data/raw/gsc/<date>.json`
   - `data/raw/ga4/<date>.json`
   - `data/raw/llm_traffic/<date>.json`
   - `data/raw/peec/<date>.json`
   - `data/processed/daily/<date>.json`
10. **Divider**
11. **Small-print footer** (quote block):
    ```
    Generated <ISO timestamp> by claude-daily routine.
    Active AI engines: <list>. Inactive (blind spots): <list>.
    URL coverage across sources: <pct>.
    Report version: <git SHA if available>.
    ```

## Weekly page body

Same MCP tool, richer content. Order:

1. **H1**: `Weekly Report – Week NN YYYY`
2. **Callout** (blue, calendar icon): Executive summary — 3-5 bullets,
   the "one thing that matters this week" first
3. **H2**: `Weekly Metrics Overview`
   - Table: metric / this week / prior week / Δ / Δ%
4. **H2**: `Trend Analysis`
   - One paragraph per metric covering direction across the 7 days
5. **H2**: `Topic-Level Breakdown`
   - Large table, one row per tracked topic
   - Columns: Topic / SEO clicks / SEO imps / GA views / GEO viz /
     GEO mentions / Cross-channel note
6. **H2**: `Winners & Losers`
   - Three mini-tables (queries / pages / topics), 5 rows each for winners
   - Three mini-tables for losers
7. **H2**: `Anomalies` (omit if zero)
8. **H2**: `Opportunity Gaps` ← REQUIRED for weekly only
   - Four subsections (H3):
     - `Rank Without Citation (GEO opportunity)`
     - `Citation Without Rank (SEO opportunity)`
     - `LLM Traffic Without Conversion (CRO opportunity)`
     - `Orphan Traffic`
   - Each with top 3 candidates (or "No items this week")
9. **H2**: `Recommended Actions`
   - 3-7 numbered items, each with owner role + due date + rationale
10. **Toggle**: `Raw data references` — all 7 daily files + weekly file
11. **Divider**
12. **Small-print footer** with data freshness + coverage metadata

## Anomaly tag vocabulary

When setting the `Anomalies` multi-select, pick from this closed list
(the DB schema enforces it):

| Tag | When to use |
|---|---|
| `Traffic spike` | Sessions or LLM sessions >2σ above baseline |
| `Traffic drop` | Sessions or LLM sessions >2σ below baseline |
| `Rank drop` | Avg position degradation >2σ or top-query position drop ≥5 |
| `Rank climb` | Avg position improvement >2σ |
| `New competitor surfaced` | A new brand appears in Peec data that wasn't there prior-week |
| `GEO opportunity` | Any candidate in the "Rank without citation" gap quadrant |
| `CRO opportunity` | Any candidate in the "LLM traffic without conversion" gap |
| `Content gap` | Any candidate in the "Citation without rank" gap |
| `Coverage gap` | Missing sources or inactive engines affecting reporting |
| `Data quality flag` | Any `severity: high` entry in `summary.geo.data_quality_flags` |

Apply liberally — multi-select is free. A weekly page with 4-6 tags is
typical; a daily page usually has 0-2.

## Update semantics

- **Daily pages**: re-running the daily routine for the same date
  **overwrites** the existing page. Claude should `notion-search` for
  `Name = "Daily Report – <date>"`, find the existing page, and use
  `notion-update-page` + block-replacement rather than creating a
  duplicate.
- **Weekly pages**: same — re-run overwrites.
- **Status field**: Claude only sets `Generated`. A human bumps to
  `Reviewed` or `Actioned` manually. If a page has `Status = Actioned`
  and Claude tries to re-run, **skip and flag** — don't silently
  overwrite actioned work.
