# Notion Setup

One-time configuration to connect this pipeline to Notion. ~5-10 minutes.

## Overview

The pipeline writes daily + weekly reports to a Notion database called
**Marketing Reports**, which lives under a parent page in your Notion
workspace (e.g. a "Marketing Analytics" or "Growth" hub page). Claude
creates the database on the first run; you only need to set up the MCP
integration and make sure it has access to the parent page.

## Prerequisites

- Access to your Notion workspace
- The Notion MCP installed and authenticated in your Claude Code session
  (Anthropic's official Notion MCP, not a third-party fork)
- Edit permission on the parent page where the database will live

## Step 1 — Verify the Notion MCP is installed

In Claude Code, open a session and ask:

```
What Notion MCP tools do you have access to? List them.
```

Expected tools (prefix `mcp__fde849c0-...`):

- `notion-search`
- `notion-fetch`
- `notion-create-database`
- `notion-create-pages`
- `notion-update-page`
- `notion-update-data-source`
- `notion-get-users`, `notion-get-teams`
- (…others)

If the MCP isn't installed, go to the Claude Code integrations panel and add
the Notion integration. Follow the OAuth flow to your workspace.

## Step 2 — Share the parent page with the integration

This is the step that most commonly gets missed and produces
"parent not found" errors.

1. Open the parent page (e.g. `https://www.notion.so/<your-workspace>/<your-parent-page>`)
   in the browser.
2. Click the `...` menu in the top-right of the page.
3. Click **Connections**.
4. In the "Connect to" modal, search for your Notion integration (usually
   called "Claude" or whatever the MCP calls itself).
5. Click to add it. The integration now has edit access to this page and
   all its children — including the Marketing Reports database we're
   about to create.

To verify from Claude Code:

```
Fetch the Notion page with ID <your-page-id> and tell me its title.
```

Should return the parent page title. If you get "not found" or "not
accessible", the integration isn't shared — go back to step 4.

## Step 3 — Configure `config/notion_schema.yaml`

Set the parent page ID in `config/notion_schema.yaml`:

```yaml
marketing_reports:
  parent_page_id: "<your-page-id>"
  parent_page_title: "<Your Parent Page Title>"
  # database_id: null  ← gets populated on the first run
```

**Do not** hand-edit `database_id` / `database_url` / `data_source_id` —
Claude writes those after creating the DB.

## Step 4 — Bootstrap the Marketing Reports database

Two options:

### Option A: let the first skill run do it automatically

Just run the daily routine. Claude reads `config/notion_schema.yaml`, sees
`database_id: null`, and runs the first-run bootstrap per the skill's
`notion-schema.md`. This creates the DB with all 14 properties and writes
the IDs back to the YAML config.

```
# In Claude Code at the repo root:
> run daily marketing report for yesterday
```

### Option B: bootstrap explicitly

```
> Read .claude/skills/marketing-analytics/notion-schema.md and execute
  the "First-run bootstrap" section for the Marketing Reports database.
```

This creates the DB without running a full daily routine. Useful if you
want to verify the schema before a long run.

## Step 5 — Sanity-check the database

After bootstrap:

1. Open the parent page in Notion
2. You should see `Marketing Reports` as a child database
3. Click in — it should show 14 properties:
   Name, Type, Period start, Period end, SEO score, GEO score,
   LLM traffic sessions, Total organic clicks, Total sessions,
   Conversions, Key insights, Anomalies, Status, Generated by,
   Source data file, Generated at

4. Confirm `config/notion_schema.yaml` now has real values:

```zsh
grep -E 'database_id|data_source_id' config/notion_schema.yaml
```

Should show something like:
```
database_id: "<your-database-id>"
data_source_id: "<your-data-source-id>"
```

## Step 6 — Create recommended Notion views

These are manual — Claude won't create them because views are editorial
and you'll want to tweak filtering/sorting to your taste. Recommended:

### View 1: "Daily dashboard"
- **Filter**: Type is Daily
- **Sort**: Period start descending
- **Gallery or board layout**
- **Visible properties**: Name, Period start, SEO score, GEO score,
  LLM traffic sessions, Key insights, Status

### View 2: "Weekly deep-dives"
- **Filter**: Type is Weekly
- **Sort**: Period start descending
- **Visible properties**: Name, Period start, Period end,
  SEO score, GEO score, Conversions, Anomalies, Status

### View 3: "Needs review"
- **Filter**: Status is Generated (not yet Reviewed/Actioned)
- **Sort**: Generated at ascending (oldest first — triage queue)

### View 4: "Trend timeline" (for the team)
- **Filter**: Type is Daily
- **Layout**: Timeline on `Period start`
- **Properties shown on cards**: SEO score, GEO score, LLM traffic sessions

## Step 7 — Teamspace or page access for colleagues

The parent page should already live somewhere your team has access to.
If someone's missing it, the fix is at the parent-page level rather than
the database level.

The **integration** (step 2) is separate from **human access**. Humans get
access through Notion's regular sharing model; the integration gets access
through its own Connections entry on the page.

## Troubleshooting

| Symptom | Fix |
|---|---|
| `parent page not found / not accessible` | Step 2 wasn't done. Share the page with the integration. |
| `cannot create database: permission denied` | Integration has "Read" access only, not "Edit". Change in Notion page → Connections → settings. |
| DB created but config YAML not updated | First-run bootstrap interrupted mid-way. Open the DB in Notion, copy the URL + data-source ID, paste into YAML by hand. |
| Page properties don't match daily-routine.md expectations | DB was created by a different tool with different schema. Delete it and re-bootstrap. |
| Daily pages created but appear outside the Marketing Reports DB | `config/notion_schema.yaml` has a wrong `data_source_id`. Re-fetch it via `notion-fetch` on the database URL. |

## Destructive-change discipline

- The skill never deletes Notion pages or the database itself.
- If a daily report for a date already exists, the skill **updates** the
  existing page rather than creating a duplicate (deterministic: looks up
  by Name like `Daily Report – 2026-04-20`).
- If an existing page has `Status = Actioned`, the skill **skips** the
  update and flags it in the reply. Human-reviewed work is never silently
  overwritten.

## If you need to recreate the database

Rare, but sometimes you want to start fresh (e.g., schema changed):

1. In Notion, navigate to the database → `...` → **Delete**. (Soft-delete —
   you can restore within 30 days from Trash.)
2. Set `database_id: null`, `database_url: null`, `data_source_id: null`
   in `config/notion_schema.yaml`.
3. Run any routine. The first-run bootstrap fires again and creates a
   fresh DB. Old reports in the deleted DB stay in Trash; the new DB
   starts empty.
